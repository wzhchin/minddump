# MindDump 蓝图 (ROADMAP)

本文件记录 MindDump 已规划但**尚未实现**的功能路线。它是 OpenSpec 的上层索引——
OpenSpec 管「正在做的一个变更」(`changes/`)与「已落地的规格」(`specs/`),蓝图管「将来要做哪些」。

- 每一项落地时,仍走 CLAUDE.md 规定的 OpenSpec 流程(提案 → 签字 → 实现 → 同步归档)。
- 蓝图条目描述**产品意图与范围**,不包含实现细节——细节在各自的提案 `design.md` 里展开。
- 优先级是相对判断(P0 最高),随版本演进调整。
- 工作量:🟢 小(≈1–2 天) / 🟡 中(≈3–6 天) / 🔴 大(≈1–2 周+)。

---

## 第一批 —— 待落地功能(本次圈选)

以下 11 项是当前规划要做的功能。**输入/采集**类先行(降低记录摩擦),
**回顾**类与**安全网**紧随。指纹相关已存在基础(`BiometricHelper`/`BiometricGate`),
本批要做的是**应用级**扩展与底层加固。

### 输入与采集

#### 1. Launcher 动态快捷方式 (App Shortcut) — P0 🟢
- **目标**:长按 app 图标弹出快捷动作——快速记文本 / 拍照 / 录音 / 直接进 Public 或 Private。
- **现状**:`ShortcutManager`/`ShortcutCompat` 零实现。
- **依赖**:无。是桌面 Widget(#1,延后)的轻量先导版。
- **关联**:为「3 秒内开始记录」这一核心诉求提供最低成本入口。

#### 2. 文本模板 / 快捷片段 (Snippet) — P1 🟢
- **目标**:一键展开预设结构(每日复盘、会议纪要、读书笔记模板)。支持新建/编辑模板。
- **现状**:`template` 零实现。
- **依赖**:模板存储(复用 DataStore / Room 轻表),不碰文件名编码。
- **关联**:降低复盘型用户的日常摩擦。

#### 3. Markdown 编辑工具栏 — P1 🟢
- **目标**:编辑器内提供粗体 / 标题 / 列表 / 代码块 / 引用等快捷插入。
- **现状**:编辑器是裸 `TextField`,内容存 `.md` 却无格式辅助。
- **依赖**:无,纯前端。作用于 `FullscreenEditScreen` 与文本输入栏。
- **关联**:既然存储格式是 Markdown,就该有匹配的输入工具。

#### 4. 导入文件夹监听 (Watch Folder) — P2 🔴
- **目标**:指定目录(截图、IM 下载),有新文件自动 import 进来,把 app 变成「个人素材黑洞」。
- **现状**:无前台服务、无文件监听。
- **依赖**:前台服务 (foreground service) + `AndroidManifest` 权限 + 通知渠道。
- **关联**:与 #11(Digest 推送)共用通知/调度基建。工程量偏大,放批次末尾。

### 关联与组织

#### 5. 标签系统 — P0 🟡
- **目标**:轻量、叠加式分类(`#idea`、`#项目A`),支持跨月份 / 跨 group 过滤与搜索。
- **现状**:`tag` 真功能零实现。
- **依赖**:**元数据走 Room 或 sidecar 文件,不再向文件名编码追加维度**
  (文件名已承载置顶 + 时间戳 + todo 状态 + role + 原名 + 扩展名 + 加密,6 维逼近极限)。
  需 Room schema 迁移策略。
- **关联**:这是后续元数据类功能(双链、自定义元数据)的**铺路石**。
  ⚠️ 收藏夹(原 #8)与之语义重叠,本批**只做标签**,放弃独立收藏夹以避免两套平行系统。

### 回顾与呈现

#### 6. 那年今天 / 时间回忆 (On This Day) — P0 🟢
- **目标**:按「历史上的今天」呈现过往记录(去年的今天、近 N 年同期)。
- **现状**:数据层已就绪(`getEntryCountByDay` 按日聚合、日历热力图),仅缺展示入口。
- **依赖**:无新增存储,复用按日查询。
- **关联**:低成本高情感价值,与 #11 共用「回顾」语境。

#### 7. 每日回顾 / Digest 推送 — P1 🟡
- **目标**:固定时间推送通知——「今日记录 N 条」「你有 M 个未完成 TODO」。
- **现状**:无通知渠道、无 `WorkManager`。
- **依赖**:通知渠道 + `WorkManager`/`AlarmManager` 调度。
- **关联**:与 #8(TODO 提醒)、#4(Watch Folder)共用通知/调度基建,适合同批落地。

#### 8. TODO 提醒(闭环已编码的 todo 状态)— P1 🟡
- **目标**:文件名已编码 `TODO/DOING/WAIT/DONE/CANCEL` 五态,却从不触发任何提醒。
  为带截止/等待语义的条目补上通知,让半成品变完整功能。
- **现状**:`reminder`/`notif`/`alarm` 零实现。
- **依赖**:通知渠道 + `WorkManager`(与 #7 共用)。
- **关联**:与 #7 同批。需要决定 WAIT 这类「等提醒」语义的具体触发模型。

#### 9. 富媒体元信息增强(EXIF / 时长 / 位置)— P2 🟡
- **目标**:照片显示拍摄时间/地点 (EXIF)、录音/视频显示时长、(可选)波形。
- **现状**:有 `file-metadata-parsing` 规格但仅解析基础元数据,未暴露 EXIF/位置。
- **依赖**:EXIF 读取库;位置信息需考虑隐私(Private 空间尤其敏感)。
- **关联**:拍摄地点可反哺 #6(按地点回忆)。

### 数据安全网

#### 10. 回收站 / 软删除 — P0 🟢
- **目标**:删除改为移入 `.trash/`(软删),保留可恢复期(如 30 天)后真删。
- **现状**:`deleteEntry()` 直接 `file.delete()` + `dao.deleteByPath()`,不可逆;
  多选批量删 (`deleteSelectedEntries`) 尤其危险。
- **依赖**:trash 目录的扫描/排除规则(纳入 `reconcileWithDisk`),过期清理调度(可借用 #7 的 `WorkManager`)。
- **关联**:**成本极低,止损极大**。与「备份」「编辑版本历史」曾并列为三大安全网,
  但后两者已改由 **git 管理兜底、明确放弃**(见文末),故回收站成为唯一需要 app 内实现的安全网。

### 安全与解锁

#### 11. 应用级生物锁 + 启动强制验证 + 密码哈希加固 — P0 🟡
> ⚠️ 指纹解锁**已部分存在**:`BiometricHelper`(BIOMETRIC_STRONG + DEVICE_CREDENTIAL)
> 已用于「切换进 Private 空间」的门禁。本项做的是扩展与加固,不是从零实现。

- **目标**:
  1. **启动强制验证**——冷启动直接进 Public,未要求验证;补「应用启动即锁」开关。
  2. **应用级锁**——当前只锁 Private 空间;提供「锁整个 app」选项。
  3. **超时重锁**——Private 解锁后会话密码常驻、无超时;补「N 分钟无操作重新加锁」。
  4. **密码哈希加固**——`PasswordStore.hashPassword` 当前用**未加盐 SHA-256**,
     易受彩虹表攻击;改为加盐慢哈希(如 PBKDF2/Argon2)并兼容旧哈希迁移。
- **现状**:生物识别门禁存在但触发点单一;密码哈希偏弱;无超时管理。
- **依赖**:设置项开关存储;哈希迁移需一次性回填逻辑。
- **关联**:与第二批的「会话密码超时」(#债务 19)重叠,本项提前覆盖其产品面。

## 第二批 —— 延后,统一补充(健壮性债务)

以下明确**不在本批**,待第一批功能落地后统一处理。列在此处以免遗漏。

### 加密与安全债务
- **流式加密**: `CryptoEngine.encryptFile/decryptFile` 整文件 `readBytes()`,
  大录像移入 Private 必 OOM。改 `CipherInputStream` 分块。
- **加密方案升级**:AES-**CBC**→**GCM**(补认证标签,可检测篡改/损坏),
  PBKDF2 迭代 10 万→60 万+。需一次性 `.enc` 迁移。
- **`.cache/` 明文残留**:解密预览/分享写明文到 `workDir/.cache`,
  靠手动 `cleanDecryptedCache`;进程被杀则残留。改用 app 私有 `cacheDir`。

## 明确**放弃**的功能(避免重复建设)

- **独立收藏夹**:语义是「命名的标签子集」,标签系统(#5)落地后放弃独立实现。
- **备份 / 导出**(zip 打包、云端同步):**改由 git 管理兜底**。
  数据目录本身即文件系统的单一真相源,用户用 git 对其做版本化即可获得备份与回溯能力,
  app 内不再重复造备份/导出/同步机制。
- **编辑版本历史** (`revision`):**改由 git 管理兜底**。
  `overwriteText` 原地覆盖造成的「改错无法回退」由 git 的提交历史覆盖,
  app 内不再实现条目级 revision。
- (其余原发散项如双链 / 未入选的 Watch Folder 等,可后续版本再评估)

> 📌 **git 兜底说明**:由于采用 git 管理数据目录,「备份」「版本历史」两项不再进入实现路线。
> 其代价是:依赖用户主动 commit/push,**误删未提交的改动 git 也救不回**——这正是 #10 回收站
> 仍需 app 内实现的原因(回收站防的是「提交前」的误删,与 git 互补,不重复)。
